笔记 Effective C++

本文最后更新于：April 16, 2022

序言

  Effective C++ 的学习笔记。

1. 让自己习惯 C++

条款 01：视 C++ 为一个语言联邦

C、Object-Oriented C++、Template C++、STL

条款 02：尽量以 const，enum，inline，替换 #define

  • 当写下 #define MY_PIE 3.14，名称 MY_PIE 也许从未被编译器看见，如果此常量产生了一个错误，这个错误信息也许会提到 3.14 而不是 MY_PIE，因为这个名称可能并未进入记号表。
  • 使用常量可能比使用 #define 导致较小量的码，因为编译器盲目地将 MY_PIE 替换为 3.14。
  • 当写下 #define CALL_MAX(a, b) f((a) > (b) ? (a) : (b))，必须记住为每个实参加上小括号，即便如此依旧会产生一些问题：

这里最好使用模板内联函数，同时保证效率和安全性。

条款 03：尽可能使用 const

  • 令函数返回一个常量值，往往可以降低用户错误造成的意外，而又不至于放弃安全性和高效性。比如这么做可以避免发生 a * b = c 这样的错误。
  • 当 const 和 non-const 成员函数有着实质等价的实现时，令 non-const 版本调用 const 版本可以避免代码重复。

注：不要让 const 版本调用 non-const 的重载版本。

条款 04：确定对象被使用前已先被初始化

  • 总是在初始化列表中列出所有成员变量，以免还须记住哪些成员变量可以无需初值。
  • 总是以声明次序列出初始化列表，以免带有初始化次序的变量们产生错误。
  • 将每个 non-local static 对象搬到自己专属的函数内，这些函数返回一个指向它所含对象的引用。
    这么做是由于：
    C++ 对于“定义于不同编译单元内的 non-local static 对象”的初始化次序并无明确定义。
    但如果以“函数调用”替换“直接访问 non-local static 对象”，就可以保证：
    对象在函数被调用时一定会被构造，不被调用时一定不会被构造。

2. 构造/析构/赋值运算

条款 05：了解 C++ 默默编写并调用哪些函数

  • 编译器可以自动为 class 创建 default 构造函数、copy 构造函数、copy assigment 操作符以及析构函数。只有当这些函数被调用，它们才会被编译器创建出来。
  • 编译器创建的 copy 构造函数、copy assigment 操作符单纯地将来源对象的每一个 non-static 成员变量拷贝到目标对象。
  • 如果一个类有数据成员不能默认构造、拷贝、复制或销毁，则对应的成员函数将被定义为删除的。
    比如含有 reference 成员、const 成员或者该类的基类将 copy 操作定义为 private 的派生类。

条款 06：若不想使用编译器自动生成的函数，就该明确拒绝

  • 为驳回编译器自动提供的技能，可以将相应的成员函数声明为 private 并且不予实现。
  • 或者继承同样操作的基类

条款 07：为多态基类声明 virtual 析构函数

  • 带多态性质的基类应当声明一个 virtual 析构函数，因为：
    当派生类对象经由一个基类指针被删除，而该基类带有一个 non-virtual 析构函数，其结果未定义。
    要消除这个问题很简单：
    给基类一个 virtual 析构函数，此后删除派生类对象便会销毁整个对象，包括所有的派生类成分。
  • 当类不企图被当作基类，令其析构函数为 virtual 往往是一个馊主意
    虚函数表指针本身会增加类的大小，而类也不再和其他语言内的相同声明拥有一样的结构。
  • 综上所述，不要继承一个带有 non-virtual 析构函数的类，比如 STL 提供的所有容器。
  • 如果希望拥有一个抽象基类，而手上又没有 pure virtual 函数，可以将析构函数声明为 pure virtual 函数，但是必须为该函数提供一份定义，否则链接器就会报错。

条款 08：别让异常逃离析构函数

  • 析构函数绝对不要抛出异常。如果一个被析构函数调用的函数可能抛出异常，析构函数应当捕获任何异常，然后吞下他们（不传播）或者结束程序。
    对 C++ 而言，在两个异常同时存在的情况下，程序若不是结束执行，就是导致不明确行为。
  • 如果客户需要对某个操作函数运行期间抛出的异常作出反应，那么 class 应该提供一个普通函数（而非在析构函数中）执行该操作。

我们用一定代价（吞下异常或结束程序）保证了析构函数不会抛出异常，同时为客户提供一个提前手动处理该异常的机会。这羊一来，如果真的触发了我们的保险，用户也没有抱怨的立场。
